# 5. Database Design

## 5.1 Overview

The TicketHub system uses **Supabase (PostgreSQL)** as the primary database. The schema consists of **20 tables** organized into 6 functional groups: User Management, Event Management, Ticketing, Shopping Cart, Orders & Payments, and Seat Map. All primary keys use `UUID` type generated by `gen_random_uuid()` unless stated otherwise. Timestamps use `TIMESTAMPTZ` (timestamp with time zone).

---

## 5.2 Table Descriptions and Relationships

### 5.2.1 `roles`

Stores the system roles assigned to users.

| Column | Data Type | Constraints | Description |
|---|---|---|---|
| `role_id` | `INT` (serial) | `PRIMARY KEY` | Auto-incremented role identifier |
| `role_name` | `VARCHAR` | `NOT NULL`, `UNIQUE` | Role name: `admin`, `seller`, `customer` |

**Relationships:**
- `roles.role_id` ŌåÉ referenced by `profiles.role_id`

| `created_at` | `TIMESTAMP` | DEFAULT `now()` | Record creation time |

### 5.2.13 `seats`

Individual seat records for events with a seat map. Supports temporary holding during checkout.

| Column | Data Type | Constraints | Description |

| `created_at` | `TIMESTAMP` | DEFAULT `now()` | Record creation time |

**Relationships:**
- `seats.ticket_type_id` ŌåÆ `ticket_types.ticket_type_id` (Many-to-One)

## 5.4 Key Design Decisions

| Decision | Description |
|---|---|
| **UUID Primary Keys** | All entities use `gen_random_uuid()` to prevent enumeration attacks and allow distributed ID generation |
| **Soft Delete on Events** | Events use `is_deleted = true` instead of hard deletion to preserve order history integrity |
| **Dual User Reference** | `carts` and `orders` carry both `customer_id` and `user_id` for backwards compatibility; `user_id` is now the primary reference |
| **Seat Hold Mechanism** | `seats.held_until` implements a 10-minute reservation window preventing double-booking during checkout |
| **Staff Token Auth** | Staff use `staff_sessions` (cookie-based token) separate from Supabase Auth, enabling offline-capable check-in |
| **Price in VND (BIGINT)** | All monetary values stored as integer VND amounts to avoid floating-point precision issues |
| **Composite PK on cart_items** | `(cart_id, ticket_type_id)` ensures one entry per ticket type per cart without a surrogate key |
| **Row Level Security (RLS)** | Every table has RLS enabled; access is enforced at the database level per user role |
